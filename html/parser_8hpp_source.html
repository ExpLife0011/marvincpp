28 /*
29  * This ABSTRACT class parses streams of data into http Message objects
30  * (or at least the first line + headers - message body is a little more complicated).
31  *
32  * This class is ABSTRACT and the expectation is that this class will be used as a base class
33  * for a more complicated object such as an incoming Request object or an incoming Response object.
34  *
35  * In homage to nodejs we could have used the name IncomingMessage - maybe we will in the future
36  *
37  * The data stream is provided to the parser using the appendBytes method. Data can be provided
38  * "all at once" - that is a complete message in one lump or "piece meal" a sequence of arbitarily sized
39  * buffers.
40  *
41  * setStreamingOption - configures the parser to parse either a single message or a (continuous) stream
42  * of messages. The default is a single message. STREAMING has not been tested
43  *
44  * Generally the parser can detect the end of a http messages as most messages formats have
45  * message length information in the message itself, or each "chunk" has a chunk length.
46  *
47  * However it is possible for a "server"
48  * to signal the end of a message by simple closing the connection. Hence in some cases the parser
49  * will only know that end-of-message has arrived when it is is told. The way to tell the parse that
50  * the end of data has been hit is to call the method appendEOF.
51  *
52  * It is therefore good practice to call appendEOF whenever the code using HTTPParser is able to detect
53  * an explicit end of data condition. There is no danger in calling appendEOF even when the message
54  * contains length infomation and the parser can self-detect the end of message.
55  *
56  * A call to parser.appendBytes will only return when:
57  *
58  * - it has processed all the bytes that it was given, or
59  * - the message had an UPGRADE method,
60  * - on of the virtual call back overrides returns 1 to pause the parser
61  * - it encountered a parse error
62  *
63  * - IT DOES NOT RETURN ON COMPLETION OF PARSING A FULL MESSAGE
64  * BUT WE HAVE FORCE THIS BY HAVING OUR MESSAGE COMPLETE CALLBACK
65  * EXPLICITLY PAUSE THE PARSER.
66  *
67  * The parser as we use it will not support reading multiple messages from the
68  * the same buffer. Each message on a connection requires an new Parser.
69  *
70  * The following "callbacks" are provided in the form of virtual methods - NOT PURE
71  * and they MAY be overridden by a derived class to capture the corresponding event
72  *
73  * virtual void OnParseBegin()
74  * virtual void OnHeadersComplete(MessageInterface* msg)
75  * virtual void OnMessageComplete(MessageInterface* msg)
76  * virtual void OnParseError()
77  * virtual void OnBodyData(void* buf, int len)
78  * virtual void OnChunkBegin(int chunkLength)
79  * virtual void OnChunkData(void* buf, int len)
80  * virtual void OnChunkEnd()
81  *
82  * The OnMessageComplete and OnHeadersComplete methods may terminate parsing by returning "true" or 1.
83  *
84  * WARNING - DO NOT PERFORM ANY IO OR RELINGUISH THE RUNLOOP/IO_SERVICE INSIDE THESE CALLBACKS
85  *
86  * PURE VIRTUAL Abstract class
87  *
88  * Derived class must implement pure virual method
89  *
90  * MessageInterface* currentMessage()
91  */
